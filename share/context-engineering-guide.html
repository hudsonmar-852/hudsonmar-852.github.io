1. 一句講清楚

Prompt Engineering 問的是：「我應該點樣問？」
Context Engineering 問的是：「AI 回答前，應該知道啲咩？」

以前很多人以為，只要寫一句很聰明的 prompt，AI 就會自然給出好答案。現在的 AI 應用愈來愈像一個小型作業系統：它會讀文件、查資料、使用工具、記住偏好、連接外部服務，甚至按步驟完成任務。

所以，真正的關鍵已經不只是 prompt，而是整個 context environment：資料來源、工具結果、記憶、規則、權限、任務狀態，以及如何把它們整理成模型最容易使用的資訊。

2. Context Engineering 五大原則

原則意思目的
Relevance
相關性		只放入與目前任務真正相關的資訊。降低干擾，避免模型被無關內容帶偏。
Sufficiency
充分性		資訊要足夠完成任務，但不需要過量。減少因資料不足而產生臆測。
Isolation
隔離性		不同任務、客戶、部門、使用者資料要分開。保護私隱、避免 context 污染。
Provenance
來源可追溯		每段資料知道來自哪裡，例如文件、API、用戶輸入或工具結果。方便驗證、引用、審計與更新。
Economy
經濟性		Token 是有限資源，只保留最高價值內容。提升速度、降低成本、減少注意力分散。

3. 與 Prompt Engineering 的分別

Prompt EngineeringContext Engineering
重點是如何寫一句好 prompt。重點是如何設計完整資訊環境。
假設模型已有足夠知識。假設模型需要動態取得外部知識。
主要優化文字表達。主要優化資料選擇、排序、壓縮、引用與權限。
多數是一次性輸入。持續管理 context 的生命週期。
像寫一條好指令。像設計一條可靠的資料管線。

簡單講：Prompt 是「最後一句話」，Context 是「整個工作環境」。

4. Context Engineering 的典型架構

User Query ↓ Task Analysis ↓ Memory Retrieval ↓ Knowledge Retrieval / RAG ↓ Tool Results / API Results ↓ Context Filtering ↓ Ranking / Compression ↓ LLM Reasoning ↓ Response + Citation + Action Log

在這個架構入面，LLM 不是唯一主角。LLM 更像最後的推理引擎，而 Context Engineering 負責決定它看到甚麼、看多少、能否追溯、是否安全。

5. 實際例子：企業內部 AI 助理

傳統 Prompt Engineering

「你是一位資深客服，請根據公司規定回答問題。」

Context Engineering 做法

  1. 接收用戶問題。
  2. 判斷問題屬於哪個部門、產品或政策範圍。
  3. 從知識庫檢索最相關文件。
  4. 只抽取必要章節，避免整份文件塞入 context。
  5. 確認資料是否足夠，不足就再次檢索或提出限制。
  6. 隔離其他客戶、其他部門或不應暴露的資料。
  7. 回答時附上來源文件、版本和日期。
  8. 把工具結果、用戶偏好及任務狀態寫入 log，方便追蹤。

這時候，真正影響回答質素的不是「你是一位資深客服」這句 prompt，而是背後的 context pipeline。

6. 如果我是你的朋友，我最想知道……

Prompt Engineering 是否已經過時？

不是。Prompt 仍然重要，但它只是一個入口。當任務愈複雜，例如寫報告、查資料、比較文件、操作 GitHub、整理 Gmail、建立知識庫，單一句 prompt 已經不足夠。

普通用戶要學 Context Engineering 嗎？

要，但不需要一開始就學到很技術。最實用的第一步是：每次叫 AI 做事前，先想清楚要給它哪些背景、限制、資料來源、輸出格式和成功標準。

企業最容易出錯的是甚麼？

把所有資料都丟給 AI，以為「越多越準」。其實太多無關內容會令模型分心，甚至引用錯資料。好的 context 是精準，不是龐大。

AI Agent 為甚麼特別需要 Context Engineering？

因為 Agent 不只是回答，它會計劃、使用工具、讀寫資料、執行步驟。只要 context 錯，它就可能用錯工具、改錯文件、發錯訊息或引用錯來源。

7. 唔講你唔知

你打的 prompt 只是其中一層

真正送入模型的內容，可能包括 system instruction、記憶、對話歷史、上載文件、工具結果、搜尋結果、圖片資訊、政策限制和用戶偏好。

MCP 像 AI 的 USB-C

Model Context Protocol 的官方說法，是讓 AI 應用連接資料來源、工具和 workflow 的開放標準。這正是 Context Engineering 走向標準化的例子。

長 context 不等於好 context

模型可以讀更多 token，不代表所有 token 都有價值。更重要是排序、壓縮、去重和來源可信度。

Memory 其實是 context 管理

ChatGPT 的記憶功能不是魔法，而是把與你有關的偏好或背景，在合適時加入 context 之中。

8. 常見誤解

誤解更準確理解
Prompt 越長越好。Prompt 要清楚，但 context 要精準。過長可能令模型分心。
RAG 就等於 Context Engineering。RAG 只是其中一部分。Context Engineering 還包括 memory、tool results、ranking、compression、provenance、permissions。
有 AI Agent 就可以全自動。Agent 更需要權限控制、檢查點、log、回滾和 human approval。
AI 答得自然就代表可信。自然語氣不等於有來源。要看引用、資料版本和是否可驗證。

9. Hudson AIOS 可用做法

如果把 Context Engineering 放入 Hudson AIOS，可以變成一套可重用流程：

  1. Project Context：每個 project 有自己的目標、規則、輸出格式和資料來源。
  2. Memory Layer：長期偏好、已完成事項、常用模板、禁用例子。
  3. Knowledge Layer：GitHub Pages、Google Drive、PDF、Report、Prompt Library。
  4. Tool Layer：Gmail、Calendar、GitHub、Canva、Web Search、Files。
  5. Filter Layer：只抽取與當前任務有關的資料。
  6. Quality Gate：檢查來源、事實、可讀性、行動性與安全。
  7. Publish Layer：輸出成 HTML、PDF、簡報或可直接貼上的內容。

最實用的升級方向：由「Prompt Library」進化成「Context Library」。

11. 三分鐘快速看完

  • Prompt Engineering 是研究如何問得好。
  • Context Engineering 是設計 AI 回答前應該看到甚麼。
  • 高質素 context 需要相關、充分、隔離、可追溯、經濟。
  • RAG、Memory、Tool Use、MCP、Agent workflow 都是 Context Engineering 的一部分。
  • AI Agent 越有能力，越需要 context 管理、權限控制和來源追蹤。
  • 未來 AI Workflow 的核心能力，不只是懂寫 prompt，而是懂建立可靠的 context pipeline。
